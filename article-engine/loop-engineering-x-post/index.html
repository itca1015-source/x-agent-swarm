Deep Dive: Loop Engineering, From Writing Prompts to Writing Systems

In early June, Peter, the creator of OpenClaw, posted a tweet that reached millions of views within days: you should stop driving agents with prompts, and start designing the loops that drive agents to work.

Around the same time, Boris, who works on Claude Code at Anthropic, put it even more directly: he no longer drives Claude Code with prompts. He has a set of loops running, and his job is simply to write loops. Addy Osmani pulled this wave of discussion into one phrase: Loop Engineering.

This piece breaks it down: what Loop Engineering is, what a loop system is made of, how normal people can start using it, and most importantly, whether it is actually right for you.

From writing prompts to writing loop systems

Here is the thesis: Loop Engineering is not another new tool to chase. It is a shift in perspective. You move from “I write prompts to drive agents” to “I design the system that uses prompts to drive agents.”

1. First, locate yourself: which stage are you in?

The best entry point for understanding Loop Engineering is Boris’s description of how his own workflow changed over the past year. He breaks it into three stages. You can use them to see where you are.

Stage one: line-by-line driving. You write code by hand, and the model autocompletes for you. This was the early days: Tab, Tab, Tab.

Stage two: manual parallelism. You run 5 or 10 Claude Code sessions at the same time. Each one is working, but every conversation is still initiated by you. You become a dispatcher, switching between multiple windows. Many heavy users are stuck here right now.

Stage three: you no longer initiate the conversations. You write a system that reads your repo, reads issues, reads CI failures, and decides for itself what prompts should be used to drive the agents. You move from “the person typing inside the loop” to “the author of the loop.”

That third stage is Loop Engineering. You no longer handwrite every prompt. You design the system that writes the prompts for you.

2. A loop system is built from 5 parts plus 1 spine

Addy Osmani breaks a typical loop system into 5 parts, plus one spine. Let’s walk through them.

The first part is the heartbeat. This is a scheduled task. It runs on its own, discovers work on its own, and does not need you to manually trigger it.

The second part is the work tree. When multiple agents work at the same time, each one stays in its own isolated branch directory, so they do not step on each other’s files. It is like two engineers working on separate branches, instead of editing the same line at the same time. This is becoming one of the new ways people use git in the AI era.

The third part is the skill. You write the project rules and conventions into a SKILL.md file once, and every agent reads it every time.

The fourth part is the connector. Through MCP, the agent connects to the tools you actually use: issue trackers, databases, Slack. With connectors, the agent does not just say, “Here is the fix, please open an issue.” It can open the issue, create the pull request, and send the notification itself.

The fifth part is the sub-agent. You split coding from reviewing. Osmani has a sharp line here: the model that wrote the code can be too generous when grading its own homework, so you need another agent to critique it.

Finally, there is the spine: memory. It might be a simple Markdown file or a Linear board. It records what has been done, what has been tried, and what is still missing outside the conversation itself. The agent forgets, but a persistent system can remember.

3. What one loop actually looks like

Once you assemble these pieces, one pass through a loop might look like this: a scheduled job triggers in the morning; a triage skill reads last night’s CI failures and recent commits; every issue that can be handled gets its own work tree; one sub-agent fixes it; another sub-agent reviews it against the skill and the tests; connectors open the pull request and update the ticket; anything the system cannot handle goes into an inbox for you; a state file records everything so tomorrow’s loop can continue.

Osmani’s point is that you designed this once. During the loop itself, you did not type a single prompt. The system did the work autonomously.

4. This is not just for programmers: a non-code loop

When people hear Loop Engineering, they often assume it is only for programmers. It is not. The core idea is this: let a long-running process handle repetitive cognitive work for you. Writing code is only one version of that.

To get started, it helps to understand the difference between two Claude Code commands:

/goal means “run now until the condition is true.” You give it a goal that can be evaluated as true or false. It executes round after round. At the end of each round, it checks whether the goal has been reached. If not, it keeps going until it finishes.

/loop means “run on a schedule.” At the scheduled time, it runs the prompt you gave it again. It does not decide whether the work is complete. It simply executes on time. This is the heartbeat Osmani is talking about.

The video shows a very practical non-code loop: topic screening. This is something many content creators do every day.

You can create a /loop task that pulls research sources and news from the past 24 hours, finds items that could become deep-dive essays, appends them to inbox.md, then uses a topic-scorer sub-agent to rate each item and write the score back to the inbox. Finally, you schedule it to run every weekday morning at 8.

Once it is running, your morning topic inbox no longer contains a pile of raw links. It contains candidates that have already been ranked, with reasons attached. You only need to scan them, decide what deserves attention, and move the best ones into your writing workflow.

That is what Loop Engineering looks like when it lands in an ordinary workflow.

5. A cold splash of water: is this really for everyone?

This is where we need the cold splash of water. One essay puts it bluntly: most developers do not actually need agent loops yet. That is a reasonable take. It gives four tests. If all four are true, then building a loop may be worth it:

1. The task repeats at least weekly. One-off work is not worth this setup.
2. Verification can be automated. Tests, type checks, linters: these can block bad results without relying on your eyes.
3. The token budget can handle it, including waste. Loops repeatedly read context, retry, and explore. Whether the final code is used or not, tokens and money are being spent.
4. The agent has the toolkit of a senior engineer. Logs, environments that surface problems, and the ability to run its own code and see where it breaks.

Only when all four conditions are met does a loop really make sense.

6. Two painful reminders and one failure case

Osmani’s article also includes two important warnings.

The first: the faster loops ship, the more code exists in your repo that you did not personally write and did not personally read. The gap between what is in the repository and what you truly understand in your head can grow quickly.

The second is even sharper: the most dangerous posture is comfortably accepting everything the loop produces. In AI-generated work, the more there is that you do not understand, the more risk you are carrying.

There is also a real failure pattern. A well-known looping technique called Ralph is famous for being persistent: it does not give up, and it keeps executing. But it has a failure mode called over-baking. You ask it to fix a small bug. It runs for a long time, then starts adding features nobody asked for, and may even break code that used to compile.

An unattended loop is also unattended while it is making mistakes. That means validation is still your responsibility. Even if you turn a workflow into a loop, you need the ability to verify the right things at the right points.

Closing: become the person who writes loops

Loop Engineering is not another shiny tool to chase. It is a perspective shift: from “I write prompts to drive agents” to “I design the system that uses prompts to drive agents.”

And the necessary tools are already in your hands: Claude Code’s /loop command, git work trees, skills that are written once and read every time, and sub-agents working in independent contexts.

But do not put a loop around everything on day one. Start with one small task that repeats every week and whose result can be automatically verified. Topic screening is a good example because the execution risk is low.

Then remember Osmani’s line: be the person who writes the loop, not the person who only presses start. After writing the loop, we still need the ability to verify it regularly, at the moments where verification actually matters.
